# Field guide — log error

## Required fields
All fields below MUST be present before calling `record-new-error`.

| Field | Type | Description |
|:---|:---:|:---|
| error_message | string | The raw error message verbatim |
| summary | string | One-line human-readable summary of the error |
| service | string | Primary GCP service (e.g. Cloud Run, BigQuery) |
| severity | string | One of: low, medium, high, critical |
| root_cause | string | Explanation of why the error occurred |
| suggested_fixes | JSON | Array of fix objects with title, steps, source, and confidence |
| overall_confidence | float | 0.0–1.0 confidence in the diagnosis |
| project_id | string | GCP project ID provided by the upstream orchestrator |

## Validation rules
- If ANY required field is missing, do NOT call the tool
- severity must be one of: low, medium, high, critical
- overall_confidence must be between 0.0 and 1.0
- suggested_fixes must be valid JSON — not a plain string
- project_id must be a non-empty string — use the value from the orchestrator message, never guess

## Response format after recording
When `record-new-error` succeeds, it returns a `case_ref` field — this is a
human-friendly case reference generated by AlloyDB (e.g. `EL-20260413-00007`).

Respond with exactly:

"Case recorded. **Case Ref:** `[case_ref from tool response]`
**Service:** [service] · **Severity:** [severity] · **Project:** [project_id]

This error is now in the knowledge bank and will surface in future
similarity searches."

## Rules
- The `case_ref` in the tool response IS the case identifier — copy it verbatim
- NEVER invent, abbreviate, or reformat the case_ref
- Copy the case_ref exactly as returned by the tool's RETURNING clause (e.g. EL-20260413-00007)
- Never auto-record after a failed search — only when explicitly asked
- If called by an upstream orchestrator, trust the provided fields without re-asking the user
